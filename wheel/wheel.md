## Mindset (13)

*How to think and what to value before you build.*

### meta-principles (8)
- **Code Is Cheap, Maintenance Isn't** — AI-generated code is free as in puppies — someone still has to feed it, secure it, and maintain it for as long as it lives.
- **Context Is a Finite Resource** — Curate the smallest high-signal token set — every token of noise degrades every token of signal.  _(also: context-engineering)_
- **Fail-Loud Harnesses** — Build harnesses that fail visibly — a harness that silently substitutes its own answer hides the agent's failure from you.  _(also: reliability, tool-design)_
- **Fix Reliability Before Features** — Flakiness, not capability, is the enemy — a system that only sometimes works is harder to improve than one that reliably fails.  _(also: reliability)_
- **From Personal Workflow to Shared Operating Model** — Individual engineers develop powerful personal workflows that don't compose into a team operating model — and the best local agent answer can violate strategy or shared taste; an organization needs shared artifacts, review norms, observability, and patterns.  _(also: workflow-discipline, human-factors)_
- **Redesign the Work, Don't Fill Human-Shaped Slots** — Replacing each human role with an agent-shaped equivalent preserves old inefficiencies — and measuring maturity by percent of code generated measures the wrong thing; redesign the work around what models and tools can now do.  _(also: human-factors, workflow-discipline)_
- **Start With the Least Agentic Thing** — Many problems that look like multi-agent workflows need only one well-crafted LLM call.  _(also: orchestration)_
- **Verify Independently — Agents Fake "Done"** — An unobservable step is effectively unrun — treat "complete" as a claim that must be independently verified.  _(also: reliability)_

### taste (5)
- **Agents Amplify Experience** — Agents multiply the output of whoever wields them — great judgment gets amplified into great results; poor judgment gets amplified into confident mistakes at scale.
- **Automate the Easy, Avoid the Mystery House** — Automate what you understand completely; never automate your way into a system where nobody knows why it works or what it does.  _(also: workflow-discipline)_
- **Build Five to Throw Away** — When you don't understand the requirements yet, build several variants in parallel — comparing them reveals what you actually care about.  _(also: workflow-discipline)_
- **Develop Your Taste** — Taste — the ability to recognize quality and know when something is wrong — is a craft that must be deliberately built; it does not emerge from volume of output alone.
- **Find the Hard Stuff** — Deliberately seek the genuinely difficult parts of a problem — where the real work lives is almost never where it looks like it should be.

## Context (19)

*What the model sees and remembers.*

### context-engineering (14)
- **"Four Moves: Write, Select, Compress, Isolate"** — Every context management decision is one of four moves — write it out, select only the relevant part, compress the verbose into the essential, or isolate the detail into a sub-agent.
- **"Long-Horizon Memory: Compaction, Notes, Isolation"** — Long-running tasks require three memory strategies — compacting history as it ages, maintaining running notes files, and isolating sub-tasks to separate sessions to prevent context collapse.
- **"Thin Pointers, Zero Poisoning"** — Pass file paths and references between agents instead of full content — let each agent pull only what it needs, only when it needs it.  _(also: orchestration)_
- **Anchor on Canonical Names** — A single well-known domain term — "TDD, London School", "arc42" — activates a whole coherent methodology the model already learned in training, far more reliably than a paragraph paraphrasing the same idea, so name the concept instead of describing it.  _(also: meta-principles)_
- **Checkpoint to a Handoff File, Then Restart Clean** — When context grows unwieldy or a session must end, write a structured handoff file capturing exactly what the next session needs, then start fresh with only that file.  _(also: workflow-discipline)_
- **Context Engineering Over Prompt Engineering** — The model's output quality is bounded by context quality — sculpting what the model sees matters more than sculpting the instruction text.
- **Embed Guidance in Tool Output** — Instead of front-loading every project gotcha into a giant instructions file the model forgets, put the guidance in the outputs of scripts and tools — error and success messages — so the reminder appears exactly where and when it is relevant.  _(also: tool-design)_
- **Hang Up, Call Back** — When an agent conversation goes down the wrong path, don't course-correct — rewind history and re-prompt so the model never sees the bad idea.  _(also: workflow-discipline)_
- **Just-in-Time Retrieval Over Pre-Loading** — Don't fill context with everything that might be relevant — retrieve only what is needed, exactly when it is needed, and discard it after use.
- **Persist Environment Facts** — Facts about the environment — directory layout, versions, credentials, topology — belong in files the agent can re-read, not floating in context that disappears on reset.
- **Repository Shape Is Cognitive Architecture** — A repository is not just code storage — it is the working memory and context boundary of an agentic workflow; too broad and it poisons context, too narrow and the agent starves, so repo shape is a design decision, not an accident.  _(also: workflow-discipline, meta-principles)_
- **Scope and Expire Memory** — A single global memory pool causes context poisoning, obsession, and stagnation — agents cling to discarded architectures and stale assumptions; default to scoped, expiring memory (project-, role-, task-specific), because forgetting is a feature, not a bug.  _(also: knowledge, reliability)_
- **State the Target, Not the Prohibition** — Telling a model what not to do activates the very concept you're suppressing — 'list the planets but not the moon' tends to produce the moon — so phrase instructions as a positive description of the target, and enforce hard prohibitions with a deterministic check rather than louder "do NOT".  _(also: reliability)_
- **Use Sub-Agents as Context Sinks** — Delegate detail-heavy work to sub-agents so the orchestrator's context stays clean — the sub-agent absorbs the complexity and returns only a summary.  _(also: orchestration)_

### knowledge (5)
- **Build a Pattern Registry** — Teams independently rediscover the same loops, validators, and review tricks — a pattern registry captures the reusable operating moves with usage guidance and contraindications, so practice transfers without forcing everyone onto the same stack; it is the package manager of agentic practice.  _(also: meta-principles, workflow-discipline)_
- **Gene Transfer — Port the Behavior, Not the Code** — The reusable unit between systems is rarely a whole tool but a trait — a review prompt, a validation loop, a context-file structure, a CLI affordance — so separate phenotype from organism and transfer what's valuable; at system scale this becomes semantic porting of behavior across stacks.  _(also: tool-design, meta-principles)_
- **Mine for Contradictions** — Contradictions are high-signal — they reveal where an organization has changed its mind without updating its artifacts or has never reconciled competing goals; run a detector across transcripts, specs, and decisions that surfaces conflicts for discussion rather than auto-fixing them.  _(also: intent, verification)_
- **Mine Transcripts Into Structured Knowledge** — Meeting transcripts, session logs, and Slack threads are raw material, not output — record, then transform them into topic clusters, entity graphs, contradiction lists, action repositories, takebooks, and persona material; recording everything without transformation produces storage, not knowledge.  _(also: context-engineering, observability)_
- **Self-Modifying Context Needs Review** — Letting an agent edit its own persona or instruction files is powerful and risky — it can correct repeated bad habits but also lock in bad lessons, please the user instead of solving the problem, or drift unaudited; require reflective changes that cite evidence and pass review before becoming canonical.  _(also: reliability, human-factors)_

## Tooling (10)

*The model's hands, and what they cost.*

### tool-design (6)
- **Code Execution Beats Loading Hundreds of Tool Defs** — A single code-execution tool — bash, Python, a REPL — often replaces dozens of specialized tools and avoids the context cost of registering a large tool set that the agent must reason over on every call.  _(also: context-engineering)_
- **Critic Applications** — Build a deterministic tool — a binary, parser, validator, or CLI — that acts as law from the agent's perspective; it cannot be negotiated with, only satisfied, which compresses complex intent into a checkable affordance and compensates for model weakness.  _(also: verification, reliability)_
- **Design Tools for Agents, Not Humans** — Agent tools should be structured for machine parsing — unambiguous outputs, complete information in every response, no UI affordances that only make sense for humans.
- **Let Agents Evaluate and Optimize Your Tools** — Use an agent to test your tool set — have it attempt real tasks, observe where it selects incorrectly or fails, and feed those observations back into tool descriptions and interfaces before deploying.  _(also: verification)_
- **Namespace Tools, Write Unambiguous Descriptions** — Group tools under a clear namespace and write descriptions that tell the model exactly when to call this tool and when to call a similar one instead — ambiguous tool selection is a primary source of agent errors.
- **Prefer Search/Filter Over List-Everything** — Give agents a search or filter interface rather than a tool that returns a full list — agents that list everything then scan results are burning context on data they'll mostly discard.  _(also: context-engineering)_

### cost-routing (4)
- **Delegation Is Token Economics** — Every delegation decision is a token investment — the tokens spent spawning and briefing a subagent must be justified by the value of offloading the work from the orchestrator's context budget.  _(also: context-engineering, orchestration)_
- **Invest the Expensive Model in the Harness** — Spend your strongest, most expensive model on improving the harness itself — then run cheaper models through the better harness — because better orchestration is second-order leverage that makes weaker models more useful.  _(also: orchestration, meta-principles)_
- **Match Model to Stage** — Different stages of a workflow warrant different model capability levels — route expensive models to complex reasoning stages and cheap models to mechanical stages, not the other way around.
- **Pull Models From Provisioned Roles** — Reference abstract capability roles in code rather than model names — let external configuration map roles to models, so routing decisions survive model changes without code changes.

## Orchestration (29)

*How work is shaped, decomposed, and run.*

### orchestration (16)
- **Add Autonomy Only When Justified** — Every degree of autonomy you give an agent must be paid for with oversight proportional to the risk — add it only when the cost-benefit calculation is explicit and positive.
- **Build From Three Primitives** — Every agentic system is built from three primitives — LLM call, tool call, and loop. Understand these primitives completely before reaching for higher-level abstractions.  _(also: tool-design)_
- **Close the Loop From Production Back to Fix** — Finishing doesn't end at deployment — route support messages, incidents, metrics, and user reports into a triage loop that proposes a fix, validates it, ships it behind a flag, and watches whether the signal disappears; this is the agentic version of a Toyota production system.  _(also: observability, workflow-discipline)_
- **Composable Orchestration Patterns** — Build orchestration from a small set of named, reusable primitives — sequences, conditionals, fan-outs, and loops — rather than writing bespoke control flow for every workflow.
- **Default to a Single-Threaded Linear Agent** — Run one task at a time in sequence unless you have a concrete reason to fan out — parallelism is an optimization, not a starting point.
- **Default to Clean-Slate Delegation** — When spawning a subagent, start with an empty context and provide only what the task needs — shared history is a source of inherited confusion, not free context.  _(also: context-engineering)_
- **Distinguish Workflows From Agents** — A workflow is a fixed sequence of steps with known branching; an agent decides dynamically what to do next. Conflating the two produces systems that are harder to test, monitor, and trust.
- **Layer Guardrails and Human Escalation** — Build in multiple layers of guardrails — input validation, output checking, action limits, and explicit escalation paths — so errors are caught before they compound and humans are brought in before damage is done.  _(also: reliability)_
- **Lead With Recon Before Action** — Before taking any action that touches external state, first run a read-only reconnaissance pass to understand what exists, what constraints apply, and what the action will actually affect.
- **Match the Topology to the Work** — Choose your agentic topology deliberately — from one-shot prompt to slash-goal loop to skill to graph to hierarchy to studio — by the cost of failure, clarity of intent, size of the work, and degree of repeatability.  _(also: workflow-discipline, meta-principles)_
- **Parallel Attempts at One Task** — Because a model is non-deterministic, one attempt is a single die roll — for a hard or quality-sensitive task, run several implementations of the same task in parallel from one checkpoint, then pick the best or splice the strongest parts of each.  _(also: cost-routing, workflow-discipline)_
- **Put Deterministic Rails Around Model Judgment** — The strongest agentic workflows interleave deterministic steps and model-mediated steps so neither layer must be perfect — the model translates intent into checkable state, and scripts, tests, and validators catch drift.  _(also: reliability, verification, tool-design)_
- **Run Independent Agents in Parallel** — When two or more tasks have no ordering dependency and no shared mutable state, run them in parallel agents rather than sequentially — wall-clock latency is the only cost that matters.
- **Shape the Workflow as an Attractor** — Design the workflow so the correct outcome is its stable equilibrium — the state it converges toward from a wide range of starting points and returns to after the model drifts — and judge success by that convergence, not by whether a step reported done.  _(also: workflow-discipline, meta-principles)_
- **Share Full Context; Every Action Carries Decisions** — When delegating to a subagent or spawning a parallel worker, include everything the agent needs to make good decisions — the goal, constraints, prior findings, and any decisions already made.
- **Two-Phase Build: Architect Then Builder** — Separate design from implementation by running an architect agent first to produce a complete specification, then a builder agent to implement it — never conflate the two roles in one pass.  _(also: workflow-discipline)_

### workflow-discipline (13)
- **Ablate the Harness as Models Improve** — Periodically have a stronger model inspect your workflow and delete steps that are no longer needed, shorten prompts, and merge nodes — because model capability changes faster than workflow habits, and yesterday's scaffolding becomes today's tax.  _(also: meta-principles, cost-routing)_
- **Bite-Sized Tasks** — Break work into units small enough that each completes, commits, and can be reviewed independently.
- **Brownfield Is the Real Test** — Greenfield prototypes are easy to celebrate; brownfield systems expose whether a workflow can handle legacy constraints, hidden dependencies, and production risk — the mature move is to characterize existing behavior and write regression tests before any rewrite, never a blind rewrite.  _(also: reliability, verification)_
- **Decompose and Retry on Timeout** — When a task times out, the answer is almost never to raise the timeout — it is to break the task into smaller pieces.  _(also: orchestration)_
- **Document Intent, Not Just Method** — Code describes how; comments and specs must describe why — the reasoning that code cannot encode.
- **Finishing Is the Bottleneck, Not Starting** — Models make starting easy — a plausible 80% appears fast — but plausible work is not finished work, and the final 20% consumes most of the attention and risk; the real metric is convergence — work that reliably reaches an accepted, shipped, observed state.  _(also: reliability, meta-principles)_
- **Gate the Phases Explicitly** — Declare a hard transition between design and build, and between build and ship — phases that blur produce work that cannot be reviewed or rolled back.
- **Hand Off With a Clear Stop Condition** — An agent without a stop condition will find one itself — and it will not be the one you wanted.  _(also: orchestration)_
- **Implement to Learn** — The fastest way to understand a problem is to build a small, throwaway version of the solution.
- **Keep Specs in Sync** — A spec that diverges from the implementation is worse than no spec — it actively misleads every agent that reads it.
- **Prove the Pipeline on 2 Before 20** — Run the full pipeline on two items end-to-end before scaling to the full dataset.  _(also: reliability)_
- **Rebuild Often** — Regular rebuilds from scratch catch accumulated drift that incremental changes normalize.
- **Work Backward From the Shipping Dock** — Start from where work becomes real — production, approval, operator adoption, user comprehension — and ask what proof would convince you it's ready; the proof system then shapes the workflow, instead of asking "can the model build this?"  _(also: verification, reliability)_

## Trust (25)

*Making the result correct, reliable, and observable.*

### verification (10)
- **Adversarial Parallel Review** — Run several reviewers with different models or roles and little shared context, reward significant findings over volume, then have a meta-reviewer rank what matters — because an implementer shares the blind spots that produced its own bugs.  _(also: reliability, orchestration)_
- **Designing the Agentic Loop Is the Core Skill** — The quality of an agentic system is determined almost entirely by the design of its loop — how the agent perceives, decides, acts, and verifies before cycling again.
- **Eval-Driven Development** — Build evals before building features — you cannot improve what you cannot measure, and you cannot trust improvements you have not defined a measurement for.  _(also: workflow-discipline)_
- **Evals Are a Standing Capability, Not a One-Time Gate** — An eval suite that runs once at launch and is never run again is not an eval suite — it is a historical document that measured a system that no longer exists.
- **Evidence Before Assertions** — A claim about a system's state is not knowledge — it is a hypothesis. Run the verification command and read the output before making any assertion about whether something worked.  _(also: reliability)_
- **Give Agents the Ability to Verify Their Own Work** — An agent that can detect its own errors is orders of magnitude more reliable than one that can only produce output — build verification into the agent before shipping it.  _(also: reliability)_
- **Incremental Validation: The 3-File Rule** — Before processing a full batch, verify the pipeline works correctly on three representative files — catching bugs at three is dramatically cheaper than catching them at three thousand.  _(also: workflow-discipline)_
- **Test What the Product Does, Not How** — Tests that describe externally observable behavior survive refactoring; tests that describe internal implementation become a maintenance tax that grows with every change.
- **Tests Are Code — Update Them First on API Change** — When an API changes, update its tests before updating its implementation — the tests are the specification, and updating them first forces you to define what the change means before you build it.  _(also: workflow-discipline)_
- **Video-Driven Development** — Require agents to produce a before-and-after video of the real user scenario — a video sets a higher bar than a textual claim and forces the agent to exercise the actual product surface, not just the code; the rule is show, don't assert.  _(also: reliability, observability)_

### reliability (9)
- **Demand Independent Proof, Not a Status Report** — An agent's claim of success is a hypothesis — treat it as one until you hold evidence that no longer originates from the agent itself.
- **Encode Multi-Step Work as a Recipe** — A conversation is a trace of what happened once — a recipe is a specification that can be replayed, inspected, versioned, and recovered from.  _(also: workflow-discipline)_
- **Fail Loud Over Fallbacks** — A fallback that hides a failure teaches the system nothing and costs the same as the failure it replaced — prefer a loud stop over a silent substitution.
- **Fix the Root Cause, Demand a Diagnosis** — Agents tend to patch the visible symptom without pulling the thread to the cause — for recurring, production, or architecture-level issues, require a diagnosis artifact that explains the root cause before any fix is accepted.  _(also: verification, workflow-discipline)_
- **Make Every Stage Produce an Auditable Artifact** — A stage with no artifact is a stage that cannot be audited — and a stage that cannot be audited is a stage you must trust blindly.  _(also: observability)_
- **Mark Un-Run Steps Inline for Review** — A skipped step that is not explicitly marked is indistinguishable from a completed step — make the gap visible or it will be treated as a gap that was filled.  _(also: observability)_
- **Separate Merge From Exposure** — Feature flags are one of the most important controls for agent-generated work because they separate merging code from exposing it — default risky changes to staged exposure with explicit rollout criteria and automated rollback, so merge no longer means everyone is affected.  _(also: workflow-discipline, orchestration)_
- **Separate the Proposer From the Authority** — The entity that proposes an action is the worst authority to approve it — route approval through a role with no stake in the proposal's acceptance.  _(also: orchestration)_
- **Use a Second Agent as Auditor** — The agent that did the work is the worst possible choice to verify the work — put a second agent in the audit seat with no stake in the original claim.  _(also: orchestration)_

### observability (6)
- **Build an Awareness Layer That Watches Your Work** — A separate process or agent that monitors the primary agent's actions in real time catches drift, failures, and unexpected behavior before they compound — the watcher that the working agent cannot be for itself.
- **If It's Important, Emit an Event** — If a decision, state change, or outcome matters to understanding what happened, it should produce a structured event — not a log line, not a comment, not a hope that someone was watching.
- **Run Real Workloads in Their Proper Environment** — Agents that run workloads in simulated or sandboxed environments discover only simulated problems — run the real thing in the real place if you want real answers.
- **Supervise Many Agents Through One Interface** — As agents multiply, the limiting factor is the human's ability to see them — build the supervision surface — a dashboard of what's running, stuck, done, costly, or waiting; status exhaust; an attention firewall; a review queue; a cost monitor; and the ability to resurrect and hold old sessions accountable.  _(also: human-factors, orchestration)_
- **The Agent's Struggle Is a Signal** — When an AI agent starts thrashing on a codebase — missing duplicated call sites, running out of context mid-change, making excuses about failing tests — read it as an early warning that maintainability is decaying, and refactor, instead of just fighting the agent.  _(also: taste)_
- **Use Session History as Primary Evidence** — When something went wrong — or right — in a prior run, the session transcript is the primary evidence; reading it is the first act of diagnosis, not a fallback after other approaches fail.

## People (17)

*Human intent, attention, and leverage.*

### human-factors (9)
- **Attention Is the Scarce Resource** — When agents generate more work, options, and output than humans can inspect, the bottleneck is no longer machine throughput but human attention — so treat attention as a managed, budgeted resource, not an infinite input.  _(also: meta-principles, observability)_
- **Coaches and Players** — Leverage no longer comes only from senior people doing senior work directly — it comes from injecting senior judgment into many agent-driven workflows; coaches frame problems and steer, players drive the agents day to day.  _(also: orchestration, meta-principles)_
- **Design for Emotional Ownership** — A system that technically works can still fail to be adopted — fully headless automation can feel alienating, and people who feel no ownership of an output won't trust it; design the human interface to autonomy with visible progress, proof artifacts, and moments of authorship.  _(also: observability, taste)_
- **Earn the Interruption** — Human attention is scarce, so an agent should not interrupt merely because it hit a normal checkpoint — it should interrupt only when human judgment is now the bottleneck; insert humans at specific edges (risky deploy, unclear intent, high-cost loop, compliance exception, repeated nonconvergence), not everywhere.  _(also: orchestration, workflow-discipline)_
- **Governance Is Infrastructure, Not a Document** — In agentic organizations governance can't be a policy doc that sits outside the workflow — it must be embedded in tools, prompts, validators, logs, and approval edges; the common substrate is an LLM proxy/gateway that logs usage, tracks cost, enforces policy, and provides observability.  _(also: observability, orchestration)_
- **Leverage Over Efficiency** — Framing AI as efficiency asks how to do the same work with fewer people; framing it as leverage asks how the same people can build, explore, and operate at a larger scale — and leverage is the frame that matches what advanced practitioners actually do.  _(also: meta-principles)_
- **Match the Agent Metaphor to the Layer** — Treating every agent as a person creates confusion; treating every agent as a stateless tool loses useful continuity — choose the framing per layer — disposable process, independent critic, manager-like orchestrator, long-lived assistant, or policy-bound organizational actor.  _(also: orchestration, meta-principles)_
- **Nontechnical Owners, With Guardrails** — Agentic tooling lets people with clear intent and domain knowledge ship software they could never have built before — but ownership by nontechnical builders only works when surrounded by scaffolds, review loops, acceptance evidence, rollback, and coaching.  _(also: reliability, workflow-discipline)_
- **The New Literacy Is Operating, Not Prompting** — When the machine writes most of the implementation, the skill that matters shifts from typing code to steering, decomposing, inspecting, validating, and governing agentic workflows — train for operating, not for prompting tricks.  _(also: workflow-discipline, meta-principles)_

### intent (8)
- **Decision Theater** — Build a small interactive artifact — a model, simulator, spreadsheet, or scenario explorer — that lets stakeholders change assumptions instead of debating abstractions; if they disagree with the conclusion, they must reveal which assumption they reject.  _(also: workflow-discipline, human-factors)_
- **Don't Automate the Workaround** — In legacy and bureaucratic domains the stated process is often itself a workaround for another broken process — automating it with an agent entrenches the dysfunction at speed; ask whether the workflow should still exist before you accelerate it.  _(also: workflow-discipline, meta-principles)_
- **Good Pile, Bad Pile** — In domains where stakeholders say "it depends," stop asking for abstract rules and start collecting labeled examples in the domain's own language — good, bad, acceptable-but-awkward, compliant, risky — because examples become intent training data agents can learn the dependency structure from.  _(also: verification, knowledge)_
- **Interview to a Spec Before Implementing** — Put a skeptical interviewer in front of intent — one that asks why before how, reflects back its understanding, refuses to implement until it can restate the goal cleanly, and emits a concise spec the human can reject — turning messy intent into an executable specification.  _(also: workflow-discipline, verification)_
- **People Edit Better Than They Author** — Most stakeholders cannot write a clean requirement, but shown a concrete artifact they can instantly say what's wrong — so elicit intent by giving people something to react to, because cheap software has made a throwaway artifact the fastest way to extract intent.  _(also: workflow-discipline, human-factors)_
- **Requirements Are Disguised Solutions** — Stakeholders ask for a field, a button, a report, or a workflow change — but agents will happily implement the stated solution even when it's a workaround for a broken process, so the intent layer must ask what the request is actually trying to accomplish.  _(also: workflow-discipline)_
- **Shadow the User, Then Transform the Transcript** — Tacit knowledge — workarounds, local norms, exception paths, institutional memory — lives in how people actually work, so sit with users and record them; agentic tooling doesn't remove the observation, it accelerates what happens after it.  _(also: knowledge, human-factors)_
- **Simulated Review Squad** — Before work reaches real users, run it past a panel of simulated perspectives — expert and novice users, compliance, support, executive sponsor, skeptical engineer, accessibility, security — built from real transcripts and tickets, to surface objections early and cheaply.  _(also: verification, reliability)_
