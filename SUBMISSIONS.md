# Submitting a Pattern

This is a curated, growing collection of patterns for building better in the agentic age. The repo is public — point your agent at this file to give it everything it needs to contribute.

## 1. Filename convention

One pattern per file, placed under `patterns/`. Name the file using the kebab-cased version of the pattern title — all lowercase, words separated by hyphens.

**THE FILENAME IS THE CANONICAL IDENTIFIER — there is no `id` field.**

| Pattern title | Filename |
|---|---|
| Leverage the Dev Factory | `patterns/leverage-dev-factory.md` |

## 2. Required frontmatter vs optional body

Every pattern file must begin with a YAML frontmatter block containing exactly three fields:

```yaml
---
title: Human-readable name of the pattern
one_liner: One sentence in plain language — this feeds INDEX.md.
dimensions: tag-one, tag-two
---
```

**Keep frontmatter flat** — three inline scalar lines only, no nested YAML.

### Optional body sections

- **Exemplars** — a markdown list where each item is formatted as `name — url — one-line note`
- **Related** — advisory cross-references to other patterns; not validated in v1

## 3. Dimension tag vocabulary

The `dimensions` field is multi-value and comma-separated. A pattern appears under **every** tag it carries.

### Seed vocabulary

| # | Tag |
|---|---|
| 1 | `meta-principles` |
| 2 | `workflow-discipline` |
| 3 | `reliability` (anti-cheating) |
| 4 | `context-engineering` |
| 5 | `tool-design` |
| 6 | `orchestration` |
| 7 | `verification` (evals) |
| 8 | `cost-routing` |
| 9 | `observability` |
| 10 | `taste` |

**Rule:** If no existing tag fits your pattern, propose a new one in the PR description. Never invent a tag silently.

## 4. Body / pearl structure

Write the body in this order:

1. **What it is**
2. **When to reach for it**
3. **When NOT to**
4. **Exemplars**
5. **Related** (optional)

## 5. One complete worked example

Below is a full pattern file you can use as a shape reference:

```markdown
---
title: Eval-Centered Development
one_liner: Write evals before writing prompts, the same way TDD writes tests before code.
dimensions: verification, workflow-discipline, reliability
---

## What it is

Eval-Centered Development (ECD) is a discipline where you define measurable success criteria for an LLM task — your "evals" — before you write or refine the prompt. The eval suite becomes the ground truth for whether a prompt change is an improvement.

## When to reach for it

- You are iterating on a prompt that drives a high-value or high-risk decision.
- You have more than one prompt candidate and no principled way to compare them.
- A prompt has regressed silently and you only noticed it in production.

## When NOT to

- The task is purely exploratory and you have no idea yet what "good" looks like.
- The output space is open-ended creative work with no agreed rubric.
- You need a quick prototype and will throw the prompt away within a day.

## Exemplars

- Hamel Husain — https://hamel.dev/blog/posts/evals/ — Practical guide to building evals before prompts
- Eugene Yan — https://eugeneyan.com/writing/llm-patterns/ — Survey of LLM engineering patterns including evaluation

## Related

- `patterns/prompt-as-code.md`
- `patterns/golden-dataset-curation.md`
```

## 6. Contribution flow

1. Fork the repository.
2. Add `patterns/{kebab-title}.md` following the conventions above.
3. Open a PR.
4. **Do NOT edit `INDEX.md`** — it is generated by `scripts/build-index.sh` and refreshed automatically by the GitHub Action on merge.
